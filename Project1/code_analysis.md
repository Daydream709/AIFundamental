# Project1 代码分析（六份）

## 1. `eight_puzzle/eight_puzzle_solvable.py`

### 功能定位

- 判断 3×3 八数码状态是否有解。
- 输入 9 个字符（`1~8` 与 `x`），输出 `1`（有解）或 `0`（无解）。

### 核心思路

- 统计除 `x` 外序列的逆序对个数。
- 对 3×3 八数码，结论是：逆序对为偶数则可达目标状态。
- 代码中 `inversion_parity` 仅返回奇偶性（`inv & 1`），更轻量。

### 复杂度分析

- 时间复杂度：$O(8^2)$，常数规模，实际非常快。
- 空间复杂度：$O(8)$。

### 优点与不足

- 优点：实现简洁，逻辑直接，适合作为求解前剪枝。
- 不足：仅适配 3×3；若扩展到 4×4（15-puzzle）需考虑空格行位置奇偶。

## 2. `eight_puzzle/eight_puzzle_min_steps.py`

### 功能定位

- 求八数码从初始状态到目标状态 `12345678x` 的最少步数。
- 无解输出 `-1`。

### 核心思路

1. 先用逆序对判定可解性，不可解直接返回。
2. 使用 BFS（广度优先搜索）按层扩展状态图。
3. `dist[state]` 记录首次到达该状态的步数；BFS 首次到达即为最短步数。
4. 用 `NEIGHBORS` 预定义每个位置的可交换下标，减少边界判断开销。

### 复杂度分析

- 状态总数上界为 $9! = 362880$，可达状态约一半。
- 时间复杂度：$O(|V|+|E|)$（在状态图语境下）。
- 空间复杂度：$O(|V|)$（队列 + `dist` 哈希表）。

### 优点与不足

- 优点：保证最短步数；实现稳定、易验证。
- 不足：只返回步数，不返回具体路径；在接近最坏情况时内存占用明显。

## 3. `eight_puzzle/eight_puzzle_astar.py`

### 功能定位

- 求八数码从起点到目标状态的操作序列（`u/d/l/r`）。
- 无解输出 `unsolvable`。

### 核心思路

1. 先用逆序对剪枝无解输入。
2. 使用 A\*：
   - $g(n)$：起点到当前状态的步数；
   - $h(n)$：曼哈顿距离启发函数；
   - $f(n)=g(n)+h(n)$。
3. 用最小堆按 `f` 取下一个扩展状态。
4. `g_score` 维护当前已知最优 `g`，并通过“过期状态跳过”避免重复无效扩展。
5. `parent[state]=(prev, action)` 记录前驱和动作，命中目标后回溯重建路径。

### 复杂度分析

- 最坏情况下仍可能接近全状态搜索。
- 实际性能通常优于纯 BFS，因为曼哈顿距离具有较好引导性。
- 空间复杂度仍为 $O(|V|)$ 量级（堆、父节点、g-score）。

### 优点与不足

- 优点：相比 BFS 通常更快；可输出完整动作序列。
- 不足：启发函数固定为曼哈顿距离，仍有可优化空间（如线性冲突）。

## 4. `shoretest_path/bfs_shortest_path.py`

### 功能定位

- 在**无权有向图**中求 1 号点到 n 号点的最短边数。
- 不可达输出 `-1`。

### 核心思路

1. 用邻接矩阵 `adj` 存图。
2. BFS 从 1 号点分层扩展，`dist[v]` 保存最短层数。
3. 当弹出节点为 `n` 时提前退出。

### 复杂度分析

- 邻接矩阵下，每次出队要扫描整行：
  - 时间复杂度：$O(n^2)$。
  - 空间复杂度：$O(n^2)$。

### 优点与不足

- 优点：逻辑直观，适合初学者理解 BFS 最短路。
- 不足：若图稀疏，邻接矩阵开销大；改邻接表可降到 $O(n+m)$。

## 5. `shoretest_path/dijkstra_naive.py`

### 功能定位

- 在**带权有向图（非负权）**中求 1 到 n 的最短路。
- 不可达输出 `-1`。

### 核心思路

1. 邻接矩阵存图，重边取最小权值。
2. 每轮在线性扫描中选取当前未确定、`dist` 最小的点 `t`。
3. 用 `t` 对所有点做松弛。

### 复杂度分析

- 时间复杂度：$O(n^2)$（选点 + 松弛都线性扫描）。
- 空间复杂度：$O(n^2)$（邻接矩阵）。

### 优点与不足

- 优点：实现稳定、代码短；在稠密图下可接受。
- 不足：稀疏图性能一般；要求边权非负。

## 6. `shoretest_path/dijkstra_heap.py`

### 功能定位

- 同样求 1 到 n 的最短路，使用优先队列优化 Dijkstra。

### 核心思路

1. 维护最小堆 `pq=(dist,node)`，每次弹出当前最短候选点。
2. `st[u]` 标记最短路已确定点，跳过重复过期记录。
3. 发现更短路径时更新 `dist[v]` 并入堆。
4. 弹出 `n` 后提前结束。

### 复杂度分析

- 当前实现仍使用邻接矩阵并遍历 `1..n` 邻居：
  - 近似时间复杂度仍偏 $O(n^2)$（虽然堆改善了选点过程）。
  - 空间复杂度：$O(n^2)$。
- 若改为邻接表，典型复杂度可达 $O((n+m)\log n)$。

### 优点与不足

- 优点：结构上已具备“堆优化 Dijkstra”的核心框架。
- 不足：数据结构与算法未完全匹配（堆 + 邻接矩阵），优化收益被限制。

## 横向对比与总结

### 八数码三题

- `solvable`：只做可解性判定，最快最轻。
- `min_steps`（BFS）：保证最少步数，但可能较慢且占内存。
- `astar`：常见输入下速度更好，且可输出操作序列。

### 最短路三题

- `bfs_shortest_path`：仅适用于无权图（或等权图）。
- `dijkstra_naive`：非负权图通用，适合稠密图/中小规模。
- `dijkstra_heap`：方向正确，但应配合邻接表才能体现明显优势。

### 可改进建议（按优先级）

1. 将 `bfs_shortest_path.py` 与 `dijkstra_heap.py` 改为邻接表存图。
2. 为八数码的 BFS 版本增加“路径重建”，便于和 A\* 结果对照。
3. 统一输入校验（字符合法性、重复元素检查）提升鲁棒性。
4. 抽取公共函数（逆序对判定）减少重复代码。
